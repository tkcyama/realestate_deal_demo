#### Supabase マイグレーション

```sql
-- 0002_purchase_offer_buyer_policies.sql
-- 買主が counter_offered → accepted/declined に UPDATE できるポリシー
CREATE POLICY "purchase_offers_update_buyer_counter"
  ON purchase_offers FOR UPDATE
  USING  (buyer_id = auth.uid() AND status = 'counter_offered')
  WITH CHECK (buyer_id = auth.uid() AND (status = 'accepted' OR status = 'declined'));

-- 買主が accepted → agreed に UPDATE できるポリシー
CREATE POLICY "purchase_offers_update_buyer_agree"
  ON purchase_offers FOR UPDATE
  USING  (buyer_id = auth.uid() AND status = 'accepted')
  WITH CHECK (buyer_id = auth.uid() AND status = 'agreed');
```

### 2026年5月6日 第3回（取引合意の二重合意バグ修正・追加整備）

#### バグ修正

| # | 対象 | 不具合内容 | 原因 | 対処 |
|---|------|-----------|------|------|
| B-15 | 取引合意 | 同一物件で別の買主と二重に取引合意できてしまう | `purchase_offers` の SELECT RLS ポリシーが `buyer_id = auth.uid()` のため、買主は他の買主の `agreed` レコードを参照できず、`agreeOffer` 内の二重合意チェックが常に空振りしていた | `0003_agreed_check_function.sql` で `SECURITY DEFINER` 関数 `check_property_has_agreed_offer()` を作成。RLS を迂回して全 `agreed` レコードを確認し、存在すればエラーを返す |
| B-16 | 取引合意 | 売主が複数の購入オファーを承諾（accepted）状態にした場合、承諾後に両方の買主が合意できてしまう | `respondPurchaseOffer` の自動見送り対象が `['pending', 'counter_offered']` のみで `accepted` が含まれていなかった | 自動見送り対象を `['pending', 'counter_offered', 'accepted']` に拡大 |
| B-17 | Tabs（オファー管理） | Base UI 警告「uncontrolled Tabs の defaultValue が変化している」がコンソールに出力される | `defaultValue` に `useSearchParams()` の値を渡していたため、SSR→CSR ハイドレーション時に値が変化してアンコントロールドからコントロールドへの切り替え警告が発生 | `defaultValue` を廃止し `value` + `onValueChange` + `useState` のコントロールド実装に変更 |

#### Supabase マイグレーション

```sql
-- 0003_agreed_check_function.sql
CREATE OR REPLACE FUNCTION check_property_has_agreed_offer(p_property_id uuid)
RETURNS boolean
LANGUAGE sql
SECURITY DEFINER
STABLE
AS $$
  SELECT EXISTS (
    SELECT 1 FROM purchase_offers
    WHERE property_id = p_property_id
      AND status = 'agreed'
  );
$$;
```

## 解決した主な技術的問題

| 問題 | 原因 | 解決策 |
|------|------|--------|
| `useSearchParams()` ビルドエラー | Next.js 16 で Suspense 必須 | ログイン・プロフィールページに `<Suspense>` ラッパー追加 |
| zod + react-hook-form 型エラー | `z.coerce.number()` が `unknown` を返す | 全数値フィールドを `z.string().refine()` に統一 |
| Supabase `.select()` API エラー | 第2引数に `count` オプション不可 | `.select('id')` のみに変更 |
| 公開申請ボタン無反応 | バリデーションエラーがコンソールのみ出力 | 上部バナーに具体的なエラー項目を一覧表示・自動スクロール |
| 地下階数バリデーションエラー | フォームにUI入力欄なく `undefined` が渡る | 地下階数の入力欄を追加（地上/構造/建築年と同列に配置） |
| 管理者の初回ログイン | 審査中のため管理画面にアクセス不可 | Supabase SQL Editor で直接 `is_admin = true` に更新 |
| フィルター関数の TypeScript 型エラー | Supabase クエリビルダーのジェネリクス型に対応する型注釈が複雑 | `applyFilters` 関数の引数を `any` 型で定義し回避 |

### Phase 3：オファー・取引フロー（完了）

#### 3-1. 型定義追加（`src/types/index.ts`）
- `SaleOffer`・`PurchaseOffer`・`Notification`・`NotificationType` 型を追加
- `SALE_OFFER_STATUS_LABELS`・`PURCHASE_OFFER_STATUS_LABELS` ラベルマップを追加

#### 3-2. Server Actions（`src/app/offers/actions.ts`）
- `sendSaleOffer` — 売主 → 買主への売却オファー送信（重複チェック・通知作成付き）
- `respondSaleOffer` — 買主による「検討する」「見送り」返答
- `sendPurchaseOffer` — 買主 → 売主への購入オファー送信（希望価格・条件・メッセージ）
- `respondPurchaseOffer` — 売主による「承諾」「条件変更」「拒否」返答
- `agreeOffer` — 買主による取引合意確定（`accepted` → `agreed` 遷移）
- `markNotificationRead` / `markAllNotificationsRead` — 通知既読管理

#### 3-3. 売却オファー送信ページ（`/offers/sale/new`）
- 物件ID（`?propertyId=xxx`）から対象物件を特定（売主本人・公開中のみ）
- 承認済み会員リストからラジオボタンで送信先を選択
- メッセージ入力・送信後はオファー一覧へリダイレクト

#### 3-4. 購入オファー送信ページ（`/offers/purchase/new`）
- `saleOfferId`（検討中オファーID）・`propertyId` をクエリパラメータで受け取り

- 購入条件・メッセージを入力して送信

#### 3-5. オファー一覧マイページ（`/offers`）
- 4セクション構成（受信売却オファー / 送信売却オファー / 受信購入オファー / 送信購入オファー）
- 未返答件数バッジ表示
- ステータス別カラーバッジ
- `SaleOfferActions`（検討する/見送りボタン）
- `PurchaseOfferActions`（承諾/条件変更/拒否、条件変更モードでは価格・条件入力欄を展開）
- `AgreeButton`（承諾後の合意確定ボタン）
- 条件変更提案の表示（counter_price / counter_conditions）

#### 3-6. 通知センター（`/notifications`）
- 全通知の一覧表示（通知タイプ別アイコン）
- 個別既読・一括既読ボタン
- サイドバーに「通知」ナビ項目を追加

#### 3-7. 物件詳細ページ権限制御強化（`/properties/[id]`）
- `sale_offer.status = 'considering'` の場合のみ収支詳細を開放（RLS連動）

- 買主向け（返答待ち）：「オファーを確認・返答する」リンク表示
- 買主向け（検討中）：「購入オファーを送る」ボタン表示（`/offers/purchase/new?...`）
