## 1. システム概要

### 1.1 プロジェクト概要

本プロジェクトは、不動産ファンド・レンダーを対象とした法人向け不動産取引プラットフォームをWebアプリケーションとして構築するものである。物件情報のアップロード・閲覧、売主‐買主間のオファー管理、レンダーへのローンオファー、AI による資料自動解析・Teaser生成、取引事例データベースの整備を通じて、不動産取引プロセスを一元管理する。

### 1.2 対象ユーザー

※ 1会員が売主・買主・レンダーの複数ロールを同時に保持可能。案件ごとにロールが切り替わる。

| ユーザー区分 | 概要 | 主な行動 |
|------------|------|---------|
| 不動産ファンド（売主） | 保有物件を売却したい会員 | 物件登録・売却オファー送信 |
| 不動産ファンド（買主） | 物件を取得したい会員 | 物件検索・購入オファー送信 |
| レンダー（金融機関等） | 不動産ローンを提供する会員 | ローンオファー受信・検討 |
| 管理者 | プラットフォーム運営者 | 会員審査・物件承認・全データ管理 |

### 1.3 システム目的・ゴール

- 不動産売買プロセスのデジタル化：物件情報・オファー・合意を一元管理
- AI活用による業務効率化：PDF/Excel/Word資料から収支・RRを自動抽出し、Teaserを自動生成
- 取引参加者のマッチング：売主が買主を指定してオファーを送るクローズドネットワーク
- レンダーとのシームレスな連携：取引合意後にローンオファーを効率的に実施
- 取引事例の蓄積・活用：成約物件のデータを業界ベンチマークとして活用

## 3. 機能要件

### 3.1 会員管理（認証・ユーザー管理）

#### 3.1.1 会員登録フロー

- Step 1：申請者がWebフォームから新規会員申請を送信（会社名・住所・部署・氏名・役職・メールアドレスを入力）
- Step 2：システムが確認メールを申請者に送信
- Step 3：管理者が申請内容を審査し、承認または却下
- Step 4：承認後、申請者にアカウント設定メール（パスワード設定リンク付き）を送信
- Step 5：初回ログイン時にロール選択（売主 / 買主 / レンダー、複数選択可）

#### 3.1.2 会員情報

| 項目 | 必須 | 備考 |
|------|------|------|
| 会社名 | 必須 | 法人名のみ。個人は対象外 |
| 住所 | 必須 | 会社本社または事業所住所 |
| 部署名 | 任意 | |
| 担当者氏名 | 必須 | |
| 役職 | 任意 | |
| メールアドレス | 必須 | ログインIDとして使用。会社ドメインのみ許可 |
| ロール | 必須 | 売主・買主・レンダーから複数選択可 |
| プロフィール（自社紹介） | 任意 | 他会員に表示 |
| 承認ステータス | システム管理 | 申請中 / 承認済 / 停止 |

#### 3.1.3 認証・セッション管理

- Supabase Auth による Email/Password 認証
- Google OAuth による SSO ログイン対応
- JWT トークンによるセッション管理（有効期限: 24時間）
- パスワードリセット機能（メールリンク方式）
- 多要素認証（MFA）対応（TOTP: 将来対応）
- 会社ドメイン制限（@example.com 等の個人メール不可、管理者設定可能）

#### 3.2.4 物件ステータス管理

```
draft（下書き）
  └─[公開申請]──→ pending_approval（承認待ち）
                      ├─[申請取り消し]──→ draft
                      └─[管理者承認]───→ published（公開中）
                                              └─[非公開に戻す]──→ draft
sold（売却完了）← Phase 5 取引完了フローで自動遷移
```

- 物件ステータスは `draft`（下書き）→ `pending_approval`（承認待ち）→ `published`（公開中）→ `sold`（売却完了）の順に遷移する
- 売主は `draft` から `pending_approval` へ「公開申請」、`pending_approval` から `draft` へ「申請取り消し」を実行できる
- 管理者は `pending_approval` の物件を `published`（承認・公開）または `draft`（差し戻し）に変更できる
- 売主は `published` の物件を `draft` に戻す「非公開」操作を実行できる。非公開後は再度編集・公開申請が可能
- `sold` への遷移は Phase 5 の取引完了フローで自動実行される。売主・管理者による手動変更は不可
- 編集操作は `draft` / `pending_approval` のステータスのみ許可する（`published`・`sold` は編集不可）

#### 3.2.5 価格・金額の入力・表示単位

| 項目 | 入力単位 | DB保存単位 | 表示単位 |
|------|---------|-----------|---------|
| 売却希望価格 | 億円（例: 50） | 円・整数（例: 5,000,000,000） | 億円（例: 50億円） |
| NOI 年間純収益 | 円 | 円 | 円・万円・億円（自動切替） |
| NCF 純キャッシュフロー | 円 | 円 | 円・万円・億円（自動切替） |
| 月額賃料収入 | 円 | 円 | 円・万円・億円（自動切替） |

- 売却希望価格はフォーム上で億円単位で入力する（例: 50 → 50億円）。DBには円単位（整数）で保存する
- NOI・NCF・月額賃料収入はフォーム上・DB保存ともに円単位で扱う
- 利回り計算（NOI利回り = NOI ÷ 価格）は、保存後の円単位の値同士で行う
- 画面表示上の売却希望価格は常に億円単位で表示する（例: 50億円・1.5億円・0.5億円）
- NOI・NCF・月額賃料収入の表示は円・万円・億円を自動切替する（formatPrice 関数準拠）

#### 3.2.6 住所正規化ルール

物件登録・編集の保存時に全住所フィールド（都道府県・市区町村・町域・番地以降）に以下の正規化を自動適用し、Geocoding API への入力ブレを防ぐ。

| 変換対象 | 変換例 |
|---------|-------|
| 全角数字・英字 → 半角 | `３９` → `39`、`Ａ棟` → `A棟` |
| 全角ハイフン・長音符 → 半角ハイフン | `３９－７` → `39-7` |
| 漢数字（単体） → 半角数字 | `三丁目` → `3丁目` |
| 漢数字（複合） → 半角数字 | `三十九番` → `39番`、`百二十三` → `123` |
| 番(地)?数字(号) → ハイフン | `39番7号` / `39番地7号` → `39-7` |
| 末尾の番地を除去 | `39番地` → `39` |
| 連続ハイフンの統一・末尾除去 | `39--7` → `39-7` |

**誤変換しないケース**: `番町`（地名）・`号館`・`号室` は変換対象外

### 3.3 地図表示機能

- Google Maps JavaScript API を使用したインタラクティブ地図
- 物件を「保有物件」「売却物件」「取引事例」の3種類に色分けしたピンで地図上にプロット
- ピンクリック → 物件カード（物件名・価格・用途・利回り）のポップアップ表示
- 地図上にカラーレジェンド（凡例）を常時表示し、ピンの種別を視覚的に識別可能にする
- フィルター機能：物件種別（保有/売却/取引事例）・用途・価格帯・利回り・エリア（都道府県・市区町村）
- クラスタリング表示：近接ピンをグループ化して視認性を向上（種別ごとにクラスター色を維持）
- リスト表示との連携：地図ピン選択でリストの該当物件をハイライト
- 表示権限制御：物件の詳細情報はオファー受理後のみ閲覧可能（地図ピン・概要カードは全会員に表示）

#### 3.3.1 地図ピンの色分け定義

| ピン種別 | ピンカラー | カラーコード | 表示対象 | 閲覧権限 |
|---------|---------|-----------|---------|---------|
| 保有物件 | 青（ブルー） | #1F3864 | 自社（ログイン中の会員）が登録・保有している物件 | 本人のみ（自社物件） |
| 売却物件 | 赤（レッド） | #C00000 | 現在売却中として公開されている物件（他社含む） | 全会員（概要のみ。詳細はオファー受諾後） |
| 取引事例 | グリーン（緑） | #1F6B3A | 成約済みで取引事例DBに登録された物件（匿名化済み） | 全会員 |

補足：同一物件が「売却物件 → 取引事例」にステータス変更された場合、ピン色は自動的に緑に切り替わる。保有物件（青）は自社登録物件のうち未公開・下書き段階のものも含む（本人のみ表示）。

### 3.4 AI自動処理機能

#### 3.4.1 収支・RR自動抽出

アップロードされたPDF・Excel・Wordファイルを Claude AI API（claude-sonnet-4-6）で解析し、以下の情報を自動抽出・登録する。

| 対象ファイル | 抽出項目 | 処理方式 |
|------------|---------|---------|
| PDF（PMR・収支表） | NOI、NCF、総収入、総費用、各費用内訳、稼働率 | Claude API（Vision + Text） |
| PDF（レントロール） | テナント名（匿名化可）、面積、賃料、契約開始・終了日、保証金 | Claude API（Vision + Text） |
| Excel（レントロール） | 上記RR項目（構造化データとして直接パース） | SheetJS / openpyxl + Claude |
| Excel（収支表） | NOI、NCF、利回り計算の基礎データ | SheetJS / openpyxl + Claude |
| Word（概要書） | 物件概要テキスト（所在地・用途・規模等） | Claude API（Text） |

注意事項：AIによる自動抽出結果は確認画面を経て登録する。誤抽出の修正は登録者が手動で行う。精度目標：主要財務項目（NOI・NCF）は95%以上の正確性を目標とする。

#### 3.4.2 物件Teaser自動生成

物件情報・アップロード資料をもとに、Claude AI API が1〜2ページのTeaserドキュメント（PDF形式）を自動生成する。

- 生成内容：物件名・所在地・概要（用途・規模・建築年）・財務ハイライト（価格・利回り・NOI）・RRサマリー・特徴コメント
- 生成フォーマット：ブランデッドPDF（HTMLテンプレート → Puppeteer → PDF変換）
- カスタマイズ：売主が公開したくない情報（テナント名等）をマスク設定可能
- 多言語対応：日本語（初期リリース）、英語（Phase 4以降）
- 生成トリガー：物件登録完了時に自動起動（バックグラウンド処理）、手動再生成も可
- 生成所要時間の目標：30秒以内

### 3.5 オファー・取引フロー

#### 3.5.1 フロー全体図

| ステップ | 内容 |
|---------|------|
| Step 1 | 売主が物件を登録（下書き → 公開申請 → 管理者承認 → 公開） |
| Step 2 | 売主が会員リストから紹介先を選んで「売却オファー」を送信 |
| Step 3 | 受信者は「検討する」または「見送り」を返信（期限: 7日間） |
| Step 4 | 「検討する」を選択した会員は物件詳細（RR・収支資料含む）にアクセス可能 |
| Step 5 | 買主会員が物件に対して「購入オファー」（価格・条件）を送信 |
| Step 6 | 売主が購入オファーを「承諾」「条件変更」「拒否」で返答 |
| Step 7 | 売主・買主が合意した場合、「合意確認」ステータスに移行 |
| Step 8 | 合意後、売主または買主がレンダー会員を選択して「ローンオファー」を送信（3.6節参照） |
| Step 9 | 取引完了後、管理者が成約情報を確認・承認し「取引事例」として登録 |

#### 3.5.2 オファー管理機能

### 3.6 レンダー機能（ローンオファー管理）

- レンダー会員に対するローンオファー送信（Step 8）：物件情報・取引合意書サマリーと共に送信
- ローンオファー受信：レンダーがオファー内容を確認し、「検討する」「見送り」を返答
- 「検討する」返答後：物件詳細・財務資料・Teaserへのアクセス権が付与される
- 融資条件のオファーバック：レンダーが融資条件（金額・期間・金利帯）を入力して返信可能
- 複数レンダーへの同時送信：売主/買主は複数のレンダーに同時にオファーを送付できる
- レンダー向けビュー：レンダー専用ダッシュボードでオファー受信状況・物件情報を一覧表示

### 3.7 取引事例データベース

- テナント名・会社名・具体的住所等の個人・法人識別情報は匿名化して公開

- 統計ダッシュボード：平均利回り・成約件数の推移・エリア別ヒートマップ（将来対応）

- CSV/Excelエクスポート機能（会員向け）

## 7. テスト計画

### 7.1 テスト方針

本プラットフォームはセキュリティ（アクセス権限制御）・データ整合性・AI処理精度が品質の核となる。以下の方針でテストを実施する。

- アクセス権限テストを最優先：誤ったアクセスによる情報漏洩は致命的なため、RLS・オファーステータス連動の権限制御を全パターン検証
- AI処理は精度テストと回復テストを実施：抽出精度の定量評価と、抽出失敗時のフォールバック動作を確認
- 取引フローはシナリオテストで検証：実際の取引ユースケースを再現したE2Eテストを全フェーズ実施
- セキュリティテストは各フェーズで実施：フェーズ完了ごとに OWASP Top 10 相当のチェック
- ステージング環境での本番同等テストを本番リリース前に必ず実施

### 7.2 テスト種別・スコープ

| テスト種別 | 目的・内容 | ツール | 実施タイミング |
|----------|---------|-------|-------------|
| 単体テスト | 各関数・コンポーネントの動作確認 | Jest / Vitest | 実装と並行（PR単位） |
| 統合テスト | API Routes + Supabase の連携確認 | Supertest / Vitest | フィーチャー完成時 |
| E2Eテスト | ユーザーシナリオのブラウザ自動テスト | Playwright | フェーズ完成時 |
| アクセス権限テスト | RLS・オファー状況によるデータ可視性検証 | Supabase テスト環境 | フェーズ完成時・回帰 |
| AI精度テスト | 収支・RR抽出精度の定量評価（テストデータ20件） | カスタムスクリプト | Phase 4 完成時 |
| パフォーマンステスト | 同時接続・レスポンスタイム計測 | k6 / Lighthouse | Phase 3・Phase 6完成時 |
| セキュリティテスト | OWASP Top 10・認証・CSRF・XSS・SQLi | OWASP ZAP / 手動 | 各フェーズ完了時 |
| 受入れテスト（UAT） | 実ユーザー（売主・買主・レンダー役）によるシナリオ検証 | 手動・テスト会 | 本番リリース前 |

### 7.3 フェーズ別テスト計画

| Phase | 主要テストシナリオ | 合否判定基準 |
|-------|----------------|-----------|
| Phase 1 | ① 会員申請〜承認〜ログインの正常フロー ② 未承認ユーザーのログイン拒否 ③ パスワードリセット ④ ロール設定と権限確認 | 全シナリオで期待動作を確認。未承認でのアクセスが100%拒否されること |
| Phase 2 | ① 物件登録〜公開申請〜管理者承認 ② 地図ピンプロット（10件・100件） ③ フィルター機能の全パターン ④ 物件詳細のアクセス権限（未オファー時は詳細非表示） | 地図表示3秒以内。権限外情報が0件表示されること |
| Phase 3 | ① 売却オファー送信〜受信〜承諾フロー ② オファー承諾後に物件詳細が閲覧可能になる ③ 見送り後は閲覧不可のまま ④ 購入オファー〜合意フロー ⑤ 期限切れオファーの自動無効化 | 取引フロー全ステップで正しい権限制御。オファー外の会員は詳細閲覧不可 |
| Phase 4 | ① PDF（10種類のPMRサンプル）のNOI/NCF抽出精度 ② Excelレントロールの全項目抽出 ③ AI失敗時のエラー通知と手動入力フォールバック ④ Teaser生成〜PDFダウンロード | NOI/NCFの抽出精度95%以上。Teaser生成30秒以内。失敗時は適切なエラー表示 |
| Phase 5 | ① ローンオファー複数レンダーへの同時送信 ② レンダーの返答〜融資条件送信フロー ③ 取引完了ステータスの正確な遷移 ④ 売却完了後の物件アクセス制限 | 全ステータス遷移が正確。取引完了後の不正アクセスが0件 |
| Phase 6 | ① 成約後の取引事例自動登録（個人情報マスク確認） ② 全会員での事例閲覧 ③ 統計ダッシュボードのデータ整合性 ④ CSVエクスポートの内容検証 | 個人情報が事例に含まれないこと（100%確認）。統計値の計算誤差0 |

### 7.4 受入れテスト（UAT）基準

| 判定カテゴリ | 判定基準 |
|------------|---------|
| 機能完全性 | 要件定義の全必須機能が動作すること |
| セキュリティ | アクセス権限外のデータへのアクセスが0件であること |
| パフォーマンス | 通常操作のレスポンスタイムが目標値以内であること |
| AI精度 | 収支・RR抽出の主要項目の精度が95%以上であること |
| 不具合 | Severity 1（システム停止・データ漏洩）が0件、Severity 2（主要機能不全）が0件 |
| ユーザビリティ | テストユーザー3名以上が主要シナリオを補助なしで完走できること |

### 7.5 バグ管理

- GitHub Issues を使用してバグを管理。Severity（1〜4）・フェーズ・担当者を必須記載
- Severity 1（サービス停止・データ漏洩）：即時対応。本番リリースブロック
- Severity 2（主要機能不全）：当該フェーズ内で修正。リリース前に全件解消
- Severity 3（軽微な機能不具合）：次フェーズ内で修正
- Severity 4（UI崩れ・軽微な表示問題）：計画的に対応
